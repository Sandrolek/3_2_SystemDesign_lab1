Теперь выполни анализ и частичное тестирование полученного прототипа TransLogix.

Твоя задача:
1. Предложить набор тестов для проверки приложения.
2. Реализовать минимум:
   - backend unit/integration tests для ключевых API
   - frontend smoke tests или component tests для критичных экранов
3. Выявить потенциальные баги, узкие места и архитектурные компромиссы.
4. Оценить качество решения по критериям:
   - функциональная полнота MVP
   - отсутствие видимых багов
   - производительность
   - читаемость кода
   - расширяемость
   - удобство развертывания
5. Подготовить материал, который можно использовать в отчёте по лабораторной.

Что проверить:
- создание и получение заказов;
- назначение заказа на маршрут;
- смена статуса доставки;
- загрузка KPI dashboard;
- low stock alerts;
- корректность seed-данных;
- ошибки API;
- работа frontend при пустых/ошибочных ответах.

Отдельно:
- оцени возможные проблемы производительности;
- оцени риски по БД, API и frontend rendering;
- предложи улучшения, которые стоит указать в рефлексивном заключении.

Формат результата:
1. таблица “что проверено / результат / комментарий”
2. список найденных проблем
3. список сильных сторон решения
4. краткий аналитический текст на 0.3–0.5 страницы для отчёта
5. краткое рефлексивное заключение от первого лица студента, но в академичном стиле
6. список возможных вопросов на защите и хорошие ответы на них

Перед тем как генерировать код:
- проверь согласованность имен сущностей и API;
- не создавай лишнюю сложность;
- ориентируйся на работающий MVP;
- если есть спор между “красиво архитектурно” и “реально запускается для лабораторной”, выбирай второе;
- в конце каждой генерации делай self-check:
  1. broken imports
  2. missing env variables
  3. mismatch between frontend and backend contracts
  4. incomplete mock data
  5. run instructions completeness
